## 사용한 기술 스택에 대해 설명해주세요.

<details>
  <summary>JWT에 대해 설명해주세요.</summary>
  JWT는 Json Web Token의 준말이고 인증을 처리할 할때 사용합니다. 2개의 토큰으로 jwt인증을
  구현하는데요. access token과 refresh token으로 관리를 합니다. 로그인과 동시에 access
  token과 refresh token을 발급한뒤, api를 호출 할때 header에 access token을 붙여서
  보내면 server에서 인증된 access token인지 확인 후 보호된 데이터를 클라이언트에게
  응답해줍니다.
</details>

---

<details>
  <summary>JWT와 쿠키,세션 로그인에 차이에 대해 설명해주세요.</summary>
  쿠키, 세션 로그인은 로그인 정보를 쿠키나 세션에 저장하는데 반해 JWT는 access token과
  refresh token 두 토큰을 이용하여 구현합니다.
</details>

---

<details>
  <summary>캐시가 뭔가요?</summary>
  자주 사용하는 데이터나 값을 미리 복사해 놓는 임시 장소이다. 캐시는 저장공간이 작고
  비용이 비싼 대신 빠른 성능을 제공한다
</details>

---

<details>
  <summary>CORS에 대해서 설명해주세요.</summary>

  허용하는 정책입니다.
</details>

<details>
  <summary>Suspense, Error boundary에 대해서 설명해주세요</summary>

  컴포넌트 대신 fallback UI를 보여주는 react component 입니다. suspense는 요번에
  정식 릴리즈가 되었고, 렌더링 할때 어떤 작업이 끝날때까지 잠시 다른 컴포넌트를 먼저
  렌더링 하는 것입니다. 한마디로 로딩시 보여줄 컴포넌트 입니다.
</details>

## 이 기술은 왜 선택했나요?

<details>
  <summary>왜 fetch말고 axios를 사용하였나요?</summary>
  Fetch는 404 에러나 다른 HTTP 에러 응답을 받았다고 해서 promise를 reject하지 않습니다.
  Fetch는 네트워크 장애가 발생한 경우에만 promise를 거부(reject) 합니다. 따라서 .then
  절을 사용해 수동으로 HTTP 에러를 처리해야 합니다. fetch는 JSON을 body에서 보낼때
  parsing이 필요한데 axios은 그 자체로 파싱을 해서 보내주고 http에러가 발생했을때도
  promise를 reject해서 더 편리해서 사용했습니다.
</details>

---

<details>
  <summary>왜 cypress를 사용했나요?</summary>
  e2e 테스팅을 하고싶어서 입니다. 한 테스트에 기능 전체를 커버할 수 있고 대부분 다시
  작성하지 않아도 되고 사용자가 앱을 사용하는 방식과 유사하기도하고 캔트 백이 통합테스트가
  투자 대비 얻는 것이 가장 많다고 했기 때문에 cypress를 사용하였습니다.
</details>

  중앙에서 상태를 관리하기 위해서 입니다.
</details>

## 기술적인 이슈를 해결했던 경험

<details>
  <summary>사내 솔루션 개발할 때</summary>
  사내 솔루션을 개발할 때 간트 차트를 교량 모델과 연동하고 시뮬레이션하는 기능을
  담당하여 개발하게 되었습니다. 간트 차트와 시뮬레이션 구현은 처음 해보는 것이었습니다.
  그래서 요구 사항들을 설계하고 어떤 것부터 구현해야 할지 분석했습니다. 요구사항들을
  작은 부분으로 쪼개어 순서대로 하나씩 해결해 나갔습니다. 먼저, 사용할 간트 차트
  라이브러리부터 분석했습니다. 그리고 차근차근 범위 슬라이더, 교량 모델과의 연동,
  시뮬레이션을 위한 interval 함수 등을 구글링하면서 스터디한 후 구현했습니다. <b>
    처음에는 막막했었으나 어떻게든 구현해야 한다는 마음가짐으로 개발
  </b>에 임했고 하나씩 쪼개어 구현하다 보니 결국 요구사항을 다 해결했을 뿐만 아니라
  기한이 남아 배속 기능까지 추가 구현을 하게 되었습니다.
</details>

<details>
  <summary>axios interceptor 에러 잡은거</summary>
  <b>원인 파악이 어려운 장애를 극복했던 경험이 있습니다.</b> jwt 인증을 구현할때
  RefreshToken과 AccessToken을 처리하기위해 axios의 interceptor를 이용했습니다. 그때
  특정된 부분이 아닌 곳에서 시간이 지나면 에러가 났었는데 도통 에러를 찾기가 굉장히
  어려웠습니다. 그래도 계속 원인을 파악하기위해 디버깅을 계속 해본 결과, 제가 달력
  데이터를 가지고 올때 이번달 가계부 데이터와 다음달 데이터, 이전달 데이터 이렇게
  동시에 3가지 api를 react-query의 prefecting으로 호출하고 있었습니다. 이것을 메세지큐를
  이용해서 토큰을 재발급 할때 큐에 호출된 api를 담아두고 새로운 토큰이 발급되면 그
  토큰을 헤더에 넣어서 다시 호출되게끔 구현을 해서 해결하게 되었습니다.
</details>

  <summary>성격의 장단점</summary>제 장점은 꼼꼼함입니다. 개발을 하면서 길러진
  장점인데, 개발을 할때 디자이너나 다른 개발자들과 협업을 자주 하게되었고 제가
  일을 대충해서 주면 맡을 일을 대충 한다는 인식과 신뢰도가 떨어진다는 것을
  느꼈습니다. 그 뒤로 최대한 꼼꼼하게 처리해서 넘겨주려고 노력을 하였고, 이러한
  습관이 개발 뿐만아니라 제 삶에 전반에서 자리잡게 되었습니다. 제 단점은 타인의
  감정을 지나치게 생각하는 것입니다. 저와 친한 분들은 감정의 선을 알아서 제가 그
  선을 넘지 않고 편하게 지낼 수 있는데, 사회적으로 만난 사람들은 그 선을 잘
  몰라서 혹시나 상처를 줄까봐 전전긍긍 했던 경험이 있습니다. 알바를 하거나
  회사를 다니거나 사회적인 활동을 하면서 사회적으로 넘지 말아야 할 선을 정의를
  했고, 그것에 의거해서 생각하려고 노력을 하면서 타인의 감정을 지나치게 생각하지
  않게 되었습니다.
</details>
